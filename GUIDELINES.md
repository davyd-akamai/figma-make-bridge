## Goal

Build a React component library, extracted from the Akamai internal ADS Figma library, that serves as the SSOT for Figma Make prototypes. What's currently published elsewhere isn't suitable for this purpose — this repo is the canonical source going forward.

## Direction: dual-purpose from the start

Components are built to work both as normal app code *and* as a package installable inside Figma Make — not sequentially, but designed in from day one, per the pattern in [Luke Finch's "How to get your design system into Figma Make"](https://finchy.medium.com/how-to-get-your-design-system-into-figma-make-3ac735205e7f), adapted to our (non-monorepo) setup.

Constraints this puts on every component:
- No app-shell assumptions — no routing, no app-specific context/providers
- `react` / `react-dom` are peer deps, never bundled
- No dynamic imports at runtime (breaks Figma Make's bundler)
- No `workspace:*` protocol deps (n/a while we're a single package, but watch for this if the repo splits)
- Styling via CSS custom properties (`var(--component-*)`) with resolved hex fallbacks — not raw Tailwind JIT classes, since Figma Make won't run our Tailwind build. Compiled `tokens.css` / `typography.css` must ship with the package, not just live in this app.

## Token architecture

Three-tier system, documented in full in [tokens/tokens-structure.md](tokens/tokens-structure.md):
`global.json` (primitives) → `light.json`/`dark.json` (semantic aliases, theme-specific) → per-component token files (`button.json`, `header.json`, etc.)

Rule of thumb: components should reference `component.*` or `alias.*` tokens, never `global.*` directly.

## Component conventions (established with GlobalHeader)

- Cite the source Figma node ID/URL in a header comment
- **Retrofit check: whenever a new shared component ships, scan already-built components for hand-rolled elements it should now replace** (e.g. Button shipping after GlobalHeader means GlobalHeader's own `<button>` markup is now stale). Don't wait for this to be requested — flag it, or do it, at the end of a new component's build. When retrofitting, prefer wrapping the shared component in a plain visibility/layout `<div>` over passing conflicting layout classes (e.g. `hidden`) straight into the shared component's own `className` — the shared component's base classes (e.g. Button's unconditional `inline-flex`) can otherwise fight a `display` override from the caller with unpredictable specificity/ordering. GlobalHeader's "Create" button was the first instance of this (retrofitted onto `components/Button.tsx` after Button shipped) — see the Button entry below.
- Responsive behavior mirrors Figma's breakpoint frames (mobile/tablet/desktop) via Tailwind breakpoint classes
- Typography via generated `.type-*` classes
- **Icons: never hand-approximate.** Every icon must be pulled as an exact vector from Figma — `download_assets` with `defaultFormat: "svg"` on the real node (or the closest exportable ancestor, e.g. a `.menu-item` instance row when the icon override itself only has a compound instance ID), then normalize coordinates into a local viewBox (subtract the icon's container offset). Approximating a shape by eye (e.g. a hand-drawn hexagon) is not acceptable even if it "looks close" — this was the exact mistake made in GlobalHeader's first draft (`MenuIcon`/`SearchIcon`/`ChevronDownIcon` were hand-drawn) and had to be redone once flagged.
- **Shared icon pack:** every icon used by any component lives in [components/icons/index.tsx](components/icons/index.tsx), one component per icon, default size 24px (matching this design system's standard icon box — `core_chevron` and `core_pin` are natively 16px and default to that instead). Components import icons from there rather than redefining them locally. Do this even for a component's first icon — don't wait until a second component needs the same icon to extract it.
- Badges (BETA/grey, NEW/violet) are a single shared [components/SystemBadge.tsx](components/SystemBadge.tsx) (renamed from `Badge` once the general-purpose [components/Badge.tsx](components/Badge.tsx) took over that name — see its entry below) matching Figma's `cm_global-badge` (node 29:3416) exactly — fixed label text and fixed colors, not a generic customizable badge. **Don't confuse this with `core_badge-reserved` (node 29:2248)** — a different, larger-specced component that isn't what's actually used inline next to labels; got this wrong once already (6px/4px padding, 12px text instead of the correct 4px/2px, 11px). When a design has two similarly-named badge-like components, check which one is actually instantiated in the row you're building (`data-name` in the fetched code) rather than assuming from the name alone.
- When a row has a trailing element (badge, chevron, pin) that should sit immediately after a label rather than pinned to the row's far edge, don't give the label `flex-1` — that stretches it to fill all remaining space and pushes the trailing element to the edge instead of leaving a clean gap. Let the label size to content and rely on the row's `gap` instead.
- Custom scrollbars (e.g. SideNavigation's menu-items list) are done via styled *native* scrollbars (`scrollbar-color` + `::-webkit-scrollbar`), not a reimplemented drag-thumb widget — same visual result as Figma's custom "Thumb" node, far less code
- When a design's token JSON file only has an ambiguous/partial mapping for a new component, extend it additively with clearly-named sibling keys (e.g. `sectionHeader`, `pin`, `scrollbar` added to `sideNavigation.json`) rather than reinterpreting or overwriting the existing ambiguous keys. If a shared component (like Badge) later needs its own tokens, give it its own token file rather than nesting it under the component that happened to use it first.
- Collapsible rails (SideNavigation): unpinned defaults to collapsed (icon-only rail), expands on hover with an eased width transition. Pinning locks the rail to a specific resting state via `pinnedExpanded` (default true) — pin doesn't only mean "locked open", it means "hover has no effect, show whatever `pinnedExpanded` says," so a permanently-collapsed-but-pinned rail is a supported, real variant, not just permanently-open. Implementation detail worth reusing: render full content (icon + label + everything) at all times at a fixed inner width, and animate the *outer* container's width with `overflow-hidden` — this clips/reveals content smoothly instead of swapping between two different DOM layouts (which would jump-cut). **Gotcha that cost a redo:** page/submenu rows must only render when `expanded && open`, never just `open` — rendering them whenever the accordion is "open" (regardless of expanded/collapsed) leaves their height in the layout even while clipped, producing visible gaps between collapsed section icons that shouldn't be there. The pin icon's filled state tracks `pinned` specifically, not transient hover-expansion — matches Figma's own Pinned=Yes/No naming (filled means "locked", not "currently visible"). A component can still take an explicit `menu="full"|"mini"` override to force a static variant with no hover/pin behavior at all (e.g. for isolated previews). Default open section (when `defaultOpenSectionIds` isn't passed) is whichever section contains the initially-selected page, not all sections — matches the reference design where only the active section is expanded by default. **Another gotcha:** any row element whose content is right/end-aligned (like the pin bar) must track the rail's *actual current width* (`w-full`, inheriting the animated parent width), not a hardcoded 232px — the fixed-232-plus-clip trick only works for left-aligned content (icons), since clipping to 48px cuts away anything positioned near the right edge of a 232-wide box. This is why the pin button disappeared entirely in the collapsed state until fixed.
- GlobalFooter: `tokens/footer.json` already existed fully-specified before this component was built (unlike SideNavigation's ambiguous partial file) — every key resolved cleanly through `light.json`/`global.json` to the exact hex the Figma frame showed, so it was used as-is with no additive extension needed. Figma's desktop frame padding read as `pt-13px`/`pb-12px` (asymmetric) — that's a sub-pixel auto-layout/text-baseline rounding artifact, not a real token, so it was normalized to symmetric `s12` (12px) top and bottom rather than inventing a non-token 13px value. `cm_link` (node 1:793, the footer's link component) was kept as a local `FooterLink` sub-component inside `GlobalFooter.tsx` rather than extracted to a shared file — extract it once a second component actually instantiates it, same bar as everything else in the shared pack.
- **Templates live in `templates/`**, a sibling to `components/`, not nested inside it — started as its own folder from the first template rather than waiting for a second one, per an explicit call on `DefaultCmPageTemplate`.
- **DefaultCmPageTemplate** (the first template) composes GlobalHeader/SideNavigation/GlobalFooter per Figma's combined `cm_page` frame (node 17:5086). That frame includes a decorative 12-column "Grid" overlay layer — Figma's own component description says to hide it before use, so it's a design-time guide, not real layout, and isn't reproduced in code. Scroll model: the page scrolls as one document, no nested overflow container. Header is plain flow (scrolls away with the page, not fixed). SideNavigation is the only fixed element — its wrapper is `position: sticky; top: 0` with height `calc(100vh - 56px)` (56px = GlobalHeader's constant height across all breakpoints), which pins it through the scroll with zero scroll-listener JS. The content column is `flex-1` inside a `min-h-screen` shell (the standard CSS "sticky footer" trick), so GlobalFooter lands at the viewport bottom on short pages and simply follows content on long ones. Below 960px, Figma's Tablet-S/Mobile-XS `cm_page` variants show SideNavigation hidden entirely (no drawer/overlay state exists anywhere in the file), so it's hidden via the same `desktop:` breakpoint already used elsewhere — no mobile nav drawer was built. The shell always composes all three sub-components (no "sidenav-less full-width" variant); consumers configure each sub-component via `headerProps`/`sideNavProps`/`footerProps` prop bags rather than the shell duplicating their full prop surfaces itself.
- **Gotcha when demoing a full-viewport template in the preview harness:** placeholder paragraph content that looks "long enough" by line count can still fit inside one viewport if the content column is wide (short sentences don't wrap at 1280px). Verifying sticky/scroll behavior needs enough real content height to actually exceed the viewport — measure rendered height, don't eyeball line count.
- Node environment note: this repo was carried over from another machine — if `npm run dev` fails with a `@rollup/rollup-*` native-binding error, it's npm's known optional-dependency bug, not a code issue; fix is `rm -rf node_modules package-lock.json && npm install`
- **Button** — [components/Button.tsx](components/Button.tsx), Figma node 47:14865 (`Button`, a local component published in "Internal ADS library" — see the Code Connect section below for why this replaced the earlier `core_button`/44:2128 citation). The kickoff prompt asked for primary/secondary/danger/link/icon, but the Figma frame only has fully-rendered swatches (all 5 states x both sizes) for **Primary/Secondary/Link/Danger**. Figma's own component description additionally lists Tertiary and Launch as types, and CDS's `ButtonVariant` separately has `'icon'` — none of those three have a single rendered swatch anywhere in the file, so per user decision they were left out of `ButtonVariant` entirely rather than guessed from tokens/CDS alone. `tokens/button.json` already has fully-specified (if unverified) entries for `tertiary`/`launch` sitting unused — extend `ButtonVariant` to include them only once real Figma swatches exist to check against. Figma names the pressed state "Active"; this file (and `tokens/button.json`, and CDS) call it `active`/"pressed" interchangeably — same state, just watch the naming when cross-referencing.
  - **Tailwind dynamic-class gotcha:** `VARIANT_STATE_CLASSES` in `Button.tsx` is written as one fully-spelled-out literal string per variant/state, not a template literal that interpolates the variant name into the CSS var path (e.g. `` `bg-[var(--component-button-${variant}-...)]` ``). Tailwind's JIT scanner only finds classes that appear as complete strings in the source — an interpolated version compiles with zero matching CSS and fails silently (no build error, just an unstyled button). This is worth remembering for any future component whose styling varies by a variant/state prop.
  - **Loading spinner** (`SpinnerIcon` in `components/icons/index.tsx`) is a static two-arc gradient SVG rotated via Tailwind's `animate-spin`, not an animated asset — pulled exact-vector from the button loading-state instances (44:2137, 44:2259) the same way every other icon is, then normalized into a 16x16 viewBox. Two color variants exist because Figma itself uses two: `variant="white"` on solid-fill buttons (primary/danger), `variant="gradient"` (green→blue) on outline/text buttons (secondary/link) — this mirrors CDS's own `spinnerWhiteIcon`/`spinnerGradientIcon` split. Gradient element `id`s are generated per-instance via `useId()` so multiple loading buttons on the same page don't collide on a shared `<radialGradient>` id.
  - **Icon color is not a separate token** — button icons (`startIcon`/`endIcon`, and the `core_add` icon reused from the shared pack via `PlusIcon`) render with `fill="currentColor"` and simply inherit the button's `text-[color:...]` class per state, matching every state in `tokens/button.json` where `.icon` and `.text` resolve to the same or near-identical color, and matching CDS's own `:host ::slotted(cds-icon) { color: inherit }`. No `--component-button-*-icon` CSS vars were added to `src/tokens.css` for this reason.
  - Icons passed via `startIcon`/`endIcon` are auto-scaled to match the button's icon size (16px large, 12px small, per Figma's "Icons automatically scale with size variants" note) via `cloneElement` injecting a `size` prop — only when the consumer hasn't already set one explicitly on the element they passed in.
  - **Retrofitted into GlobalHeader** immediately after shipping: the "Create" CTA (previously a hand-rolled `<button>` using `alias.action.primary.*` tokens directly) now renders `<Button variant="primary" size="large" startIcon={<PlusIcon />}>Create</Button>`. Visually identical (those alias tokens resolve to the same hex `component.button.primary.default/hover` do), but confirmed via computed-style inspection rather than assumed. The mobile/tablet-hidden behavior (`hidden tablet:flex`) was moved onto a wrapping `<div>` rather than appended to Button's own `className`, per the retrofit-check convention above.
- **Container** — [components/Container.tsx](components/Container.tsx), Figma node 58:5033 (`Container`, a local component published in "Internal ADS library" — see the Code Connect section below for why this replaced the earlier `core_container [not coded]`/41:2135 citation). The node's own `[not coded]` layer name was literal: `get_metadata`/`get_design_context`/`get_variable_defs` all agreed it's a bare, non-auto-layout box with zero children — only `background`/`border` resolve to real variables (`{alias.background.normal}`/`{alias.border.normal}`), no padding/radius/size data exists on the node itself. `tokens/container.json` (already present before this build) turned out to be byte-identical to CDS's own `container.json` — same two ambiguous color-only keys — and CDS has no matching Lit `container` component either (only a much larger, differently-named `content-panel`), confirming this is genuinely a minimal primitive, not an under-fetched one. **Padding (24px horizontal / 16px vertical) was confirmed directly by the user rather than sourced from the node** — mapped to the existing `s24`/`s16` global spacing scale, applied straight in Tailwind classes (`px-[var(--global-spacing-s24,24px)] py-[var(--global-spacing-s16,16px)]`) rather than added to `tokens/container.json`, matching Button's precedent that spacing lives in component markup, not in the color-only per-component token JSON files. No radius, fixed size, or variants exist anywhere in the file, so the component is intentionally just `background` + `border` + that padding + a `children` slot — a generic bordered surface, not a card/panel with header/footer regions.
  - **Gotcha for future "[not coded]" Figma nodes:** don't assume a bare/empty node means "nothing to build" or invent structure (padding, radius, header slots) to fill the gap — stop and ask. This one turned out to need a real (if undocumented-in-Figma) padding value that only the design owner could supply.
  - **Full-width-by-design, not content-sized:** per the design owner, a Container always occupies the entire available width of its parent grid/content area — it never sizes to its children. Implemented as an explicit `w-full` in the component's own class list rather than relying on default block-level `width: auto`, since `auto` only happens to fill available width in normal block flow and would silently shrink-to-fit if a consumer ever placed a Container inside a flex/grid row without an explicit stretch. This behavior (and the "never place two side by side" corollary) is documented in `figma-make-guidelines.md`'s Container section and Known-constraints list so Figma Make's AI doesn't try to size or row-place it like a card.
  - **Real-world usage confirmed** against a NAT Gateway details screenshot from an actual Akamai Cloud prototype: two Containers stacked full-width, each holding independently-composed content (a title+Edit/Delete header row, a grey "SUMMARY" section-label bar, a label/value grid; a second Container holding a sortable/expandable table). Confirmed with the design owner that this composed content is out of Container's own scope — Container stays a plain bordered box + `children`, and the header row/section bar/table are the consumer's job to compose (the table specifically previews what the upcoming **Table** component in the roadmap needs to support: sortable columns, an info-icon column header, and expandable/nested rows).

- **TextField** — [components/TextField.tsx](components/TextField.tsx), Figma node 47:7385 (`core_text-field`). Figma demos an 8-state x 3-label-position matrix (Default/Placeholder/Hover/Focus/Filled/Disabled/Error/Read only x None/Top/Left) — nearly all of it collapsed onto native browser/CSS behavior instead of JS-tracked state: `hover:` and `focus-within:` Tailwind variants on the field wrapper reproduce Hover/Focus, and Placeholder-vs-Filled is just the browser's own placeholder-vs-value rendering (verified border/text/placeholder colors are literally identical between e.g. `hover` and `hoverEmpty` in the token file — only the browser-native content differs, not the styling). The only real React state is `value` (controlled/uncontrolled split, same `value ?? internalValue` pattern as SideNavigation's `selectedPageId`), needed to know when to show the clear button.
  - **`tokens/textField.json` already existed, pre-seeded and fully specified** (same "Jan 12" batch as `button.json`/`header.json`/`menuButton.json`) — but it didn't show up in an initial `ls tokens/` at the very start of this build's session (worth re-checking directly by filename, not just grepping file contents, before assuming a token file is missing). Every color in it was cross-checked against `get_variable_defs` and the actual exported icon SVGs' own fill colors, and all matched exactly — including a split this file got right that a first-pass manual read of `get_variable_defs` almost missed: the close/clear icon (`#3D3D42`, `alias.content.icon.primary.default`) and the info icon (`#696970`, `alias.content.icon.secondary.default`) are two different colors, confirmed by the literal `fill="var(--fill-0, ...)"` in each icon's downloaded SVG, not the same "icon color" token guessed from the close icon alone.
  - **Two new shared icons**: `InfoIcon` (`core_info`, node 47:7392, 20px) and `CloseIcon` (`core_close`, node 47:7480/47:7390, 16px), added to [components/icons/index.tsx](components/icons/index.tsx). **Gotcha:** `download_assets` on the icon's own node ID returned `null` for `core_info` — its containing "Info icon" frame is `hidden="true"` in the Default-state variant (the specific instance `get_metadata` pointed at), and `download_assets` respects per-node hidden state so it can't export a hidden node. The fix was pulling the icon from the image URLs `get_design_context` had already generated inline (`imgCoreInfo`, `imgCoreClose`, etc., at the top of its returned code) — that tool renders every image reference across the whole requested subtree regardless of any individual node's hidden flag, so those URLs worked where a second, node-scoped `download_assets` call on the same icon did not.
  - **Label position**: `labelPosition?: "top" | "left"`, default `"top"` — per user decision, following Figma's own written best-practice guidance ("should include a visible label on top, most cases") over CDS's Lit default of `"left"`. Figma's third variant, "Label position=None", isn't a separate enum value here — it's just the natural result of omitting the `label` prop entirely (no label text, nothing to position), which avoids a redundant prop combination (`label` set but `labelPosition="none"`, or vice versa).
  - **Info icon is a deliberate "basement" for a future Tooltip**, per user decision — this repo has no Tooltip/Popover component yet (Figma's own description says the icon "triggers tooltip/popover with additional context"). `infoIcon`/`infoText`/`onInfoIconClick` render the icon as a real `<button>` with a native `title` (browser tooltip) and a click callback for now; a real Tooltip component can replace the `title` with an actual popover trigger later without changing this prop surface.
  - **Read-only renders a structurally different tree**, not just a style variant — no border/background/input element at all (matches CDS's `:host([readonly]) { background: transparent; border: none }` and Figma's own padding data, where the readonly row's left padding drops to `0` vs. every other state's `s12`). The label is a plain `<span>` (not `<label htmlFor>`) since there's no focusable control left to associate it with once the `<input>` is gone — `<label for>` pointing at a non-form-control span isn't meaningful to assistive tech, so the value text gets `aria-label` directly instead.
  - **Typography reused wholesale from `src/typography.css`** — `.type-label-bold-s` (field label), `.type-label-regular-s` (typed value), `.type-label-regular-placeholder` doesn't end up used directly (placeholder styling is done via Tailwind's `placeholder:italic` since `::placeholder` inherits the input's own font metrics but not `font-style`), `.type-label-semibold-xs` (helper/error text) — all four turned out to already exist as exact matches for this component's four Figma text styles (Label/Bold S, Label/Regular S, Label/Placeholder S, Label/Semibold XS), so no new typography classes were needed.
  - **Clear (×) button only appears when `clearable` (default `true`) and the field has a value and isn't disabled** — matches CDS's own `clearable` getter (`Boolean(this.value) && !this.password`). Its icon color has its own independent `hover:`/`active:` states (`--component-textfield-icon-hover`/`-active`, resolving to brand blue) layered on top of the field's own hover/focus states, since the token file specifies a `hover`/`active` triplet on the icon color distinct from the field border color.
  - **Bug found via user review, fixed same day:** the info icon was first built nested *inside* the bordered field box, as a third trailing element alongside the input and clear button. A separate Figma callout the user pointed to (node 47:14252, "Text field variants") makes clear it's actually a **sibling of the box**, in its own `shrink-0` slot outside the border — matches the `get_design_context` output for 47:7385 too (the "Field & info icon" wrapper always had "Field elements" and "Info icon" as siblings; the nesting-inside-the-box was a misread on the first pass, not a real ambiguity). Fixed by wrapping `[box (flex-1), infoIconButton (shrink-0)]` in an outer flex row instead of rendering the icon as a third child of the box itself.
  - **Bug found via user review, fixed same day:** hovering over a *focused* field (blue border) briefly flashed the hover border color (near-black) instead of staying blue — `hover:` and `focus-within:` are separate Tailwind utilities with equal CSS specificity (0,1,1 each), so when both conditions are true at once (mouse resting on a focused field), whichever rule Tailwind happens to emit later in the compiled stylesheet wins, which isn't something to rely on. Fixed with a compound arbitrary variant, `[&:hover:not(:focus-within)]:border-...`, which has higher specificity than `:focus-within` alone *and* simply doesn't match at all while focused — so focus deterministically wins regardless of generation order. **General pattern worth reusing:** whenever two pseudo-class variants on the same element are meant to be mutually exclusive with one taking priority (not just "last one wins"), guard the lower-priority one with `:not()` rather than trusting Tailwind's variant emission order.

- **Checkbox** — [components/Checkbox.tsx](components/Checkbox.tsx), Figma node 58:4716 (`Checkbox`), internal breakdown node 58:4359 (`.base-checkbox`). Native `<input type="checkbox">` visually hidden (`sr-only`) inside a `<label>`, with a custom-drawn box as its sibling — real keyboard/click/form semantics for free, exact-vector visuals layered via CSS, same pattern CDS's Lit `cds-checkbox` uses.
  - **`size` prop (`"small" | "large"`, default `"large"`) — added despite the composed Figma component not exposing it.** The actual instantiated `Checkbox` node (with label) hardcodes `size="Large"` in its own generated code, no prop at all. But the internal `.base-checkbox` primitive it's built from has fully-specified Small (16px)/Large (20px) swatches for every state, and CDS's own `cds-checkbox` ships a real `size: 'small' | 'medium'` property backed by the same tokens. Unlike Button's Tertiary/Launch (zero rendered swatches anywhere, correctly left out), Small here has full visual + CDS-behavioral coverage — different enough evidence bar to include it. Both sizes share the same 12×12 checkmark/dash icon; only the box size and padding (`s2` vs `s4`) change.
    **Usage rule (user decision, 2026-07-09):** `"large"` is the default and covers almost every checkbox on a page; `"small"` is reserved specifically for tables and dense list components where a 20px control would crowd the row — not a general-purpose compact variant. Documented for Figma Make's AI in `figma-make-guidelines.md`'s Checkbox section and its Known-constraints list, same treatment as GlobalHeader's `showSearch`/`logo` being flagged as fixed surface rather than a per-prototype knob.
  - **Figma's "State=Focus" variant resolves through token keys literally named `active`** (e.g. `component/checkbox/empty/active/border`), not `focus` — the token *bucket name* is the inconsistent part, not the Figma variant label (mirrors Button's Figma-says-"Active" naming quirk, just the mismatch is flipped here: variant name is correct, bucket name is off). No separate mouse-press-only state exists anywhere in the file. CDS's own Lit source resolves the ambiguity for us: it uses the same `active`-bucket tokens for both real `:active` (mouse press) and `:focus-visible` (keyboard focus ring) — see `checkbox.styles.ts`'s `:has(input:focus-visible)` rule. Implemented the same way here via a `peer-focus-visible:` Tailwind variant targeting the hidden input, no JS-tracked focus state.
  - **`indeterminate` is a plain prop, not derived from `checked`** — set on the real DOM `indeterminate` property via a `useRef`/`useEffect` (React has no JSX attribute for it, same as vanilla DOM). It's purely visual and consumer-controlled, same as CDS: clicking does *not* auto-clear it internally, so a "select all" parent is expected to recompute `indeterminate={someChecked && !allChecked}` itself on every render (demoed in the preview harness's select-all pattern). This matches native browser behavior only loosely — native indeterminate auto-clears on user click, but a React-controlled prop re-asserts itself next render regardless, so auto-clearing inside the component would just fight the consumer's own state.
  - **Disabled and read-only are both real `disabled` on the native input** (`nonInteractive = disabled || readOnly`), not just a visual treatment — matches CDS's `?disabled=${this.disabled || this.readonly}`. They render distinct box classes (grey fill vs. transparent-with-thin-border) rather than sharing one `:disabled`-driven style, since Figma's disabled and read-only tokens differ structurally (readOnly has no `background` key at all in `tokens/checkbox.json`, only `border`), not just by opacity.

A rebuild that's instant and synchronous on a fast machine can still race a browser's async input-targeting pipeline, especially for elements the compositor may handle on their own layer (`position: fixed`, portaled to `document.body`). Prefer touching only the specific DOM nodes that actually changed.
    - **Process lesson, arguably the more important one:** when a control test's conclusion conveniently absolves your own code, that is exactly the moment to distrust it hardest, not least. ## First real Make swap-in: what broke and why (2026-07-09)

First incremental swap (shell + Button + Container into the existing Routing Table prototype, which predates this library and had its own hand-rolled Tailwind setup already in place) surfaced a real bug, not just doc gaps:

- **Symptom:** `GlobalHeader`/`SideNavigation` rendered their sub-960px layout (hamburger, no search bar, no logo, sidenav fully hidden) even at a confirmed ~2190px viewport.
- **Root cause, confirmed via a debug panel Make added at our request (`document.styleSheets` + a recursive `element.matches()` walk over every rule, since `@media`/`@layer` blocks nest and a flat `cssRules` read misses them):** the prototype's own pre-existing Tailwind output defines a plain `.hidden { display: none }` rule with no media query, alongside `figma-make-bridge`'s correctly-defined `@media (min-width: 960px) { .desktop\:flex { display: flex } }`. Both rules have equal specificity (single class selector each), so **the one loaded later in the document wins** — and the app's own stylesheet was loading after ours.
- **Interim fix:** moved the `figma-make-bridge/styles.css` import to load *after* the app's own stylesheet, so ours wins the cascade tie.
- **Known tradeoff of that fix, not a separate bug:** this flips cascade precedence *globally*, not just for the one `.hidden` rule — any class name our compiled output happens to share with the app's own pre-existing Tailwind build (`.flex`, `.gap-4`, `.rounded`, etc. are common names) now resolves to our token values instead of the app's original ones, even on markup we never touched. Expect cosmetic drift in untouched areas after this fix; it's a known consequence of import-order being a blunt, global lever.
- **Durable follow-up (not yet done):** give `figma-make-bridge`'s Tailwind build a **class prefix** (e.g. `fmb-`) so its compiled utilities can never collide by name with any other Tailwind build again — removes the whole bug class instead of relying on import order, which is fragile (can flip again on the next Make regeneration and isn't something we control). Requires regenerating the compiled CSS with a prefix and updating every component's className strings to match. Sized as a real task, not done inline during this debugging session.
- **Separate, environment-level gotcha (not a library bug):** returning to a Make file after a day+ idle sometimes shows `Can't resolve 'figma-make-bridge/styles.css'` on startup — Make's own diagnosis was a container reset wiping `node_modules` between sessions while the lockfile/symlink stayed intact; reinstalling the dependency (which Make's own "resolve" flow does automatically) was the only fix needed, no package-side change. Expected friction of an ephemeral dev container, not something to chase further on our end.

## Public component preview site (`docs/`, built 2026-07-10)

User wanted a publicly-visible page showing every migrated component/template, one page each, components and templates kept separate — built as a static HTML/CSS/JS site with no React and no build step (not the same thing as `src/App.tsx`'s dev-only harness, which requires `npm run dev`), so it's viewable by anyone via GitHub Pages without cloning the repo.

- **Structure:** `docs/index.html` (landing page, Components/Templates card grids) + `docs/components/*.html` (one per component) + `docs/templates/*.html` (one per template) + `docs/assets/` (shared `styles.css`, `site.css`, `shell.js`, plus per-component `*.js` interaction scripts). Mirrors the `components/` vs `templates/` split in the source tree, per the user's explicit ask to keep them separate.
- **Styling reuses the real compiled output, not a re-guessed approximation:** `docs/assets/styles.css` is a straight copy of `npm run build`'s `dist/styles.css` (must be regenerated and re-copied after any component/token change — nothing automates this yet). Every demo's markup uses the literal Tailwind arbitrary-value classes and `--component-*`/`--alias-*`/`--global-*` CSS custom properties copied verbatim from the corresponding `.tsx` source, so colors/spacing/typography always trace back to the same tokens the real React components render — never hand-picked hex values.
- **Interactivity is real, not static screenshots** (explicit user choice over a cheaper static-swatch-sheet approach): Button's variant/size/loading/icon playground, Checkbox's indeterminate + select-all-parent recompute pattern, RadioButton's native `peer-checked:` grouping, TextField's controlled clear-button, and SideNavigation's full hover-expand/pin-lock/accordion state machine are all hand-reimplemented in vanilla JS in the matching `docs/components/*.js` file, mirroring each component's actual class-selection logic (e.g. `Button.tsx`'s `VARIANT_STATE_CLASSES`) literal string-for-string rather than reinventing the styling logic.
- **Gotcha that cost a debug pass: a stray `*/` inside a CSS comment silently ate the entire `:root` variable block.** `site.css`'s opening comment originally contained the substring `component.*/alias.*/global.*` — the `*/` mid-sentence closed the comment early, and the browser then parsed the rest of the comment text as a garbage selector that swallowed the real `:root { --docs-bg: ...; }` rule into its (unmatched) declaration block. Symptom was every `var(--docs-*)`-based style silently resolving to nothing (e.g. active toolbar buttons rendering with no background) with zero console error, since invalid-selector rules just don't match rather than throwing. Caught by walking `document.styleSheets` in the browser and diffing actual parsed rules against source. **Lesson: never let literal `*/`-shaped text appear inside a CSS comment, even mid-sentence.**
- **Gotcha: `desktop:`/`tablet:` are real viewport media queries, not container queries** — an early draft tried to demo GlobalHeader/GlobalFooter's mobile layout by constraining a plain narrow `<div>` wrapper, which does nothing since the media query still evaluates against the actual browser window width. Fixed by rendering the forced-breakpoint demos inside `<iframe>`s instead — each iframe gets its own real viewport, so setting the iframe's own `width` genuinely changes which breakpoint applies, independent of the parent page/browser width. Used for GlobalHeader's mobile/tablet/desktop rows, GlobalFooter's forced-narrow demo, and the whole `DefaultCmPageTemplate` live preview (also needed its own real scroll container to demo the sticky SideNavigation behavior).
- **`DefaultCmPageTemplate`'s live preview iframe is pinned to 1040px wide** (with `overflow-x:auto` on its wrapper) rather than `width:100%` — at the docs page's actual content-column width (~1020px after page padding), a 100%-wide iframe sometimes lands just under the 960px desktop breakpoint *inside* the iframe, which hides SideNavigation entirely (correct component behavior, but defeats the point of the demo). Fixing the width above the breakpoint guarantees the interesting (desktop, sidenav-visible) state renders by default.
- **Local preview:** `.claude/launch.json` runs `npx serve docs -l 4173` (plain `python3 -m http.server` failed sandboxed with a `PermissionError` on `os.getcwd()` in this environment — worth remembering if that command is tried again elsewhere).
- **Publishing:** GitHub Pages, `main` branch, `/docs` folder — needs to be turned on once in the repo's Settings → Pages (not something a file change alone can do). README's Components table links to `https://davyd-akamai.github.io/figma-make-bridge/...` per component, live once Pages is enabled.
- **Automated (2026-07-10):** [.github/workflows/sync-docs-styles.yml](.github/workflows/sync-docs-styles.yml) runs `npm run build` on every push to `main` that touches `components/**`, `templates/**`, `tokens/**`, `src/**`, or `vite.lib.config.ts`, diffs the result against `docs/assets/styles.css`, and commits+pushes the refreshed copy back to `main` if it changed. Deliberately **not** triggered by `docs/**` itself, so the bot's own sync commit doesn't re-trigger the workflow. Uses the default `GITHUB_TOKEN` to push directly (no PR) — if `main` ever gets branch protection requiring reviews, this will need to switch to opening a PR instead.
  - **Bug this caught immediately:** Tailwind v4's automatic content detection scans the whole project respecting `.gitignore`, and `docs/` isn't gitignored — so a plain rebuild after `docs/` existed picked up utility classes from the *preview site's own demo HTML* and leaked them into the published package's `dist/styles.css` (148.3KB vs. the correct 147.5KB). Fixed with `@source not "../docs";` in [src/library.css](src/library.css), right after the `@import "tailwindcss"` — confirms the library build only ever reflects real component source, never the docs site's own markup, regardless of what class names happen to appear there.
